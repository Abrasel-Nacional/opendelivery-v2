    Authentication and Authorization¶

    This section defines how participants authenticate and obtain access to protected resources and operations in The Open Delivery Protocol (ODP).

    Authentication is not a capability.

    It is a protocol-wide mechanism that accompanies capability usage and provides a common access model across participants.

    Normative Rules¶

    1. Implementations MUST support a merchant-scoped credential model.
    2. Implementations MUST NOT share one credential set across unrelated merchants.
    3. Authentication material MUST be protected in transit and at rest.
    4. Access scopes MUST be explicit and auditable.
    5. Event callbacks MUST use an authenticated trust mechanism.
    6. Token acquisition flow MUST be defined consistently for all protected ODP operations.

    How Authentication Relates to Capabilities¶

    Authentication exists to protect resources and operations exposed by capabilities such as Merchant, Orders, and Logistics.

    It does not represent a standalone business coordination area.

    Some protocol functions may be public, but most productive interactions happen only after the participant is authenticated and authorized.

    Operation-Level Authentication Declaration¶

    Each capability operation should explicitly document:

    • Whether the operation is public or protected
    • Which scope is required when the operation is protected
    • Which participant is expected to present the credential
    • Which authentication trust rule applies to callbacks or asynchronous notifications

    This declaration should appear in the operation definition itself, with a reference back to this section.

    Authentication Context Fields¶

    name type required description
    credential.ownerRole string YES Participant role that owns the credential (ORDERING_APPLICATION, SOFTWARE_SERVICE, LOGISTICS_SERVICE)
    credential.merchantId string YES Merchant scope for the credential
    credential.clientId string YES Client identifier
    credential.clientSecret string YES Client secret
    credential.scope string YES Granted protocol scope
    credential.expiresAt string NO Credential expiration timestamp (ISO 8601 date-time)
    credential.rotationPolicy object NO Rotation policy metadata

    Authentication Context Example¶

    {
  "credential": {
    "ownerRole": "ORDERING_APPLICATION",
    "merchantId": "merchant-123",
    "clientId": "oa-merchant-123",
    "clientSecret": "***",
    "scope": "od.all"
  }
}

    Conformance Notes¶

    • The authentication and token exchange model is part of the protocol and is shared across ODP implementations.
    • Callback trust verification remains required even when the primary operation flow is asynchronous.
